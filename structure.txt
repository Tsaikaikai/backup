AOI 視覺檢測平台系統架構設計報告1. 核心設計理念本平台採用 「介面導向開發 (Interface-Driven)」 與 「中央狀態機控制 (State Machine Control)」 為核心。透過將硬體行為抽象化，並由中央引擎統一調度，確保系統具備高穩定性、易測試性與快速更換硬體的靈活性。2. 三層架構職責定義 (Three-Tier Architecture)I. UI Layer (展示層)技術棧： WPF + MVVM。核心任務： 數據視覺化、操作指令下達、即時影像顯示。實務規範： * ViewModel 僅透過 Command 觸發 Core 層的狀態切換。嚴禁直接引用任何相機 SDK 或演算法 DLL。透過 Data Binding 同步 Core.Engine 中的生產數據與狀態。II. Core App Layer (邏輯核心層)核心任務： 狀態機運作、生產流程 (Flow) 調度、HAL 介面定義、配方 (Recipe) 管理。關鍵組件：Main Engine: 運行主狀態機，負責從一個狀態轉移到另一個狀態。Inspection Session: 封裝「單次檢測」的所有數據（影像、結果、標籤、時間戳）。HAL Interfaces: 定義 ICamera, ILighting, IAlgo, ICommunication 等抽象介面。實務規範： 所有的業務邏輯（例如：取像後多久要觸發演算法）皆在此層實現。III. Infrastructure Layer (基礎設施實作層)核心任務： 具象化硬體操作、第三方庫封裝。關鍵組件：Hardware Drivers: 如 HikCamImpl, ModbusTcpPLC。Algorithm Wrappers: 封裝 AI 推論引擎或 OpenCV 算子。Data Services: 實作 LogRecorder 與 DatabaseRepository。實務規範： 此層僅負責「執行指令」，不參與邏輯判斷。3. 狀態機系統設計 (State Machine Design)系統運作邏輯由 有限狀態機 (FSM) 驅動，確保硬體動作與軟體邏輯在任何時刻都是唯一且可預測的。狀態說明：狀態 (State)說明觸發條件 (Transition)Resetting系統啟動中，進行硬體連線、參數加載、演算法初始化。程式開啟 / 從 Error 恢復。Idle所有硬體已就緒，等待生產指令或 UI 按下 Start。初始化成功。Running核心取像流程（Trigger -> Grab -> Inspect -> Output）。接收到開始訊號（UI 或 PLC）。Paused暫停生產流，但保持硬體連線與參數掛載。UI 按下 Pause 或 門禁觸發。Error/Alarm硬體故障、通訊逾時或演算法異常。Try-Catch 捕獲例外 / 硬體斷線。4. 關鍵工作流程 (Inspection Workflow)當系統處於 Running 狀態時，執行以下自動化循環：Wait Trigger: 通訊模組監聽 PLC 訊號。Acquisition: 呼叫 ICamera.Grab()，取得影像並產生一個 InspectionSession。Process: 將影像丟入 IAlgorithm.Execute() 進行非同步推論。Result Update: 彙整演算法結果，更新至 UI 並同步寫入 Local DB。Feedback: 透過 ICommunication 將 OK/NG 訊號回傳給 PLC/機械手臂。Archiving: 異步執行存圖任務，避免影響下一週期取像。5. 實務作法與團隊規範 (Standard Operating Procedures)A. 介面與實作分離 (Interface Segregation)開發人員 A 在 Core 層定義 ICamera 介面。開發人員 B 在 Infrastructure 層寫 BaslerCamera 實作。優點： 如果相機缺料要換品牌，只需新增一個實作類別，Core 層代碼一行都不用改。B. 虛擬化開發 (Mocking)在 Infrastructure 中務必提供 VirtualCamera 與 VirtualPLC。這讓工程師在辦公室或家裡即可模擬完整的自動化流程，無需綁定實體硬體。C. 異步任務處理 (Async/Await)UI 與 Engine 運行在不同執行緒。所有的 I/O 操作（存圖、寫資料庫、MES 上報）必須採用 Fire-and-Forget 或 Task Queue 模式，嚴禁阻塞檢測主循環。結語本架構在「簡潔性」與「擴展性」之間取得了平衡。對 1-2 人的團隊而言，這能極大化開發效率，同時為未來可能的系統升級（例如：增加多站點聯網、導入更複雜的 AI 流程）打下堅實的基礎。架構師簽核： [你的名稱/Gemini]日期： 2026-03-30


狀態 (State),進入條件 (Entry),允許操作 (Operations),離開條件 (Exit)
INITIALIZING,程式啟動,自動檢測硬體連線、版本檢查,全部硬體回傳 OK -> IDLE
IDLE,初始化完成 / 停止作業,載入 Recipe、進入 Maintenance、修改系統參數,Recipe 載入成功 -> READY
READY,Recipe 已掛載,Start (開始生產)、解除 Recipe,點擊 Start -> RUNNING
RUNNING,收到 Start 指令,執行自動檢測流程、Pause、Stop,點擊 Stop 或 流程結束 -> READY
PAUSED,執行中按下 Pause,查看當前 Session 數據、Resume (繼續)、Abort,點擊 Resume -> RUNNING；Abort -> READY
MAINTENANCE,IDLE 時切換至工程模式,手動調試硬體 (單動相機/馬達/IO),退出模式 -> RESETTING
RESETTING,故障排除後 / 維修退出,硬體歸零 (Homing)、清空 Buffer、狀態復位,復位完成 -> IDLE
ERROR,任何層級拋出 Exception,診斷錯誤碼、Acknowledge (確認),點擊 Reset -> RESETTING
SHUTTING_DOWN,關機指令,關閉硬體連線、儲存最後狀態、寫入 Log,資源釋放完成 -> 關閉程式

子狀態 (Sub-State),職責說明,Timeout 處理機制
WAITING,等待 PLC 或 軟體 Trigger 訊號,超過設定時間未觸發 -> 拋出 Warning
IMAGING,開啟光源、相機取像、檢查影像完整性,取像失敗 / 超時 -> 進入 ERROR (相機斷線)
PROCESSING,影像前處理、演算法推論 (CNN / 幾何匹配),演算法運算超時 -> 標記 JUDGE_TIMEOUT
JUDGING,綜合各站結果、判定 OK/NG、與 Recipe 規範比對,判斷邏輯異常 -> 預設給予 NG 安全機制
REPORTING,更新 UI、寫入本地 DB、上報 MES、存圖 (Async),上報超時 -> 寫入本地緩存 (Cache) 後繼續
TRANSFERRING,送出實體 OK/NG 訊號給 PLC、等待產品離開,PLC 未回應完成訊號 -> 進入 ERROR (塞料)
